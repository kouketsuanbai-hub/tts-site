## 決定（2026-06-16）

進め方は段階方式を採用する。

1. まず案A（自分用ツール＋ダメ元広告）で公開し、土台（デプロイ・AdSense・自分用の実利）を確保する。
2. 公開後、PV と使われ方を一定期間観察する。
3. その結果を見て、案B（収益本気狙い）へ載せ替えるか判断する。

なぜこの方式か
- 案A→案B は機能・構成の足し算で地続き。いま二択で決め切る必要がない。
- 案Aは実働 半日〜1日・ランニングコストほぼゼロで、損が出ない。
- 案Bの本質コスト（記事 20〜30時間）を、需要が確認できるまで先送りできる。

収益の前提認識（重要）
- 手間をかけずに月数万円は困難。現実的中心は月0〜数千円。
- 案Bを回しても現実的中心は月数千〜2万円台、上振れで月2〜3万。月5万は低手間前提では到達困難。
- 一方でランニングコストはほぼゼロ（Cloudflare 無料＋静的）なので赤字リスクは無い。

## 分析サマリー（2026-06-15 の多角分析より）

| 観点 | スコア | 要点 |
|---|---|---|
| 技術実現性 | 3/5 | PC では堅実に作れて壊れにくい。iOS Safari は画面ロック・バックグラウンドで音声停止（Apple の仕様で回避不能）＝「スマホでながら聞き」は破綻 |
| 競合優位性 | 3/5 | 強い競合は無料・低品質ゾーンに降りてこない（手薄）。だが同型個人サイトはゼロコストで量産されうる。堀はSEOの場所取り頼み |
| 収益性 | 2/5 | 月数万円には月10〜25万PV必要。収益になるのはツール画面でなくSEO記事のPV |
| 市場需要 | 3/5 | 需要は実在（「テキスト読み上げ」月約4,400検索）。だが ChatGPT 本体・OS標準読み上げが同用途を無料で吸収中 |

収益試算（AdSense：必要PV ＝ 目標額 ÷ RPM × 1000）
- 実測アンカー：日次1000PV（月3万PV）≒ 月1万円
- 月2万円 → 月8〜13万PV、月5万円 → 月20〜33万PV
- ツール特性で RPM は低位（50〜250円）に沈みやすく、必要PVはさらに膨らむ

3つの構造的な逆風
1. 収益は SEO 記事のPV依存 → 「手間ゼロ」と両立しない
2. ChatGPT 本体＋OS標準の読み上げが主用途を外側から侵食している
3. 新規個人サイトは今の Google で「インデックスされてもランクしない」のが現実的着地

## 案A：自分用ツール＋ダメ元広告版

狙い：自分が毎日使う道具を公開し、広告はダメ元で貼る。収益はおまけ。

コンセプト・看板
- 「テキスト読み上げ（汎用）」のまま。用途は絞らない。
- iOS Safari はバックグラウンド/ロックで止まるため、注意書きを1行足す（「スマホは画面を表示したまま」）。

サイト構成（現状から増やさない）
```
index.html / about.html / privacy.html
assets/（style.css / tts.js / favicon.svg / og.png）
robots.txt / sitemap.xml
```
広告は index のツール下に1枠、about 記事内に1枠。AdSense 審査の最低ライン（about/FAQ/privacy 併設）は満たすが、単機能ゆえ一発合格は保証されない。

収益期待値：月0〜数千円。手間・コストはほぼゼロで赤字なし。

## 案B：収益本気狙い版（案A＋集客設計）

狙い：月数千〜数万円を本気で狙う。

1. 用途のリポジション（最重要）
- 最弱点の iOS ながら聞きを看板から外し、技術的に支えきれる土俵へ。
- 主訴求：「PCで・長文を・サーバーに送らず・耳で読む / 校正する」
- description 例：「ChatGPT の長文や原稿を、サーバーに送らずブラウザだけで読み上げ。登録不要・文字数無制限。PC での校正・ながら確認に。」
- スマホは切り捨てない。画面を開いている間は普通に使えるので、モバイル流入は歓迎し、UI もレスポンシブを維持する。看板にしないのは「画面ロック・バックグラウンドのまま聞く」用途のみ（iOS の仕様で停止するため、そこだけ期待値を作らず正直に書く）。

2. コンテンツ拡充（ロングテール記事 5〜10本／`/guide/` 配下）

| 記事タイトル案 | 狙うキーワード |
|---|---|
| ChatGPT の回答をコピペで読み上げる方法 | ChatGPT 読み上げ |
| 長文を耳で読む・校正する無料のやり方 | 長文 読み上げ 無料 |
| テキストを送信せず読み上げる（機密文書でも安全） | テキスト 読み上げ 送信しない |
| 登録不要で使える無料テキスト読み上げまとめ | 読み上げ 無料 登録不要 |
| ブラウザの読み上げ音声を増やす・変える方法 | 読み上げ 音声 追加 |
| 文章を音声で校正するコツ（PC） | 文章 校正 読み上げ |
| 英語テキストを読み上げて発音確認する方法 | 英語 読み上げ 無料 |
| Web Speech API とは（仕組みとプライバシー） | Web Speech API |

3. SEO強化
- 各記事に meta/OGP、構造化データ（Article / FAQPage）、内部リンク（記事⇄ツール）
- sitemap を記事分まで拡張、パンくず
- ツール本体に FAQ 構造化データを追加

4. 差別化機能の磨き込み（PC前提）
- 選択再生・続き再生をランディングで前面に言語化
- 速度プリセット（校正用ゆっくり / 確認用速め）、段落単位ジャンプ
- 「依存ゼロ・シンプル」は崩さない範囲で

5. 収益導線
- AdSense 主体（記事ページに配置。ツール画面は見られにくいので過度に期待しない）
- アフィリは慎重に。「無料・送信なし」を看板にしながら有料・送信型TTSを薦めるのは矛盾するため、隣接物販（イヤホン・読書グッズ等）か矛盾しないものに限定。

6. 手間とタイムライン
- 初動1〜2ヶ月：記事5〜10本（1本2〜4時間 × 8 ≒ 20〜30時間）＋SEO設定
- 以降は放置可。月1回の仕様チェック
- 3ヶ月後：日次数十〜数百PV、月数百〜2,000円
- 1年後：記事が当たれば日次1,000〜3,000PV、月3,000〜2万円台。上振れで月2〜3万

収益期待値：現実的中心 月数千〜1万円台、上振れで月2〜3万。月5万は低手間前提では到達困難。

## リサーチ追補（2026-06-16）

詳細：[技術・広告リサーチ_2026-06-16.md](技術・広告リサーチ_2026-06-16.md)

競合認識の修正
- 本当の競合はスマホアプリ（棒読み/ゆっくり系）ではなく OS標準の読み上げ機能。中身は同じ端末内蔵 TTS で音質に差がつかない。
- 棒読みちゃんは Windows 専用・更新停止。ゆっくり/AquesTalk 系はキャラ音声という別ジャンルで継続リスクあり。素の読み上げでは直接競合にならない。
- 優位性の置き場所を「対アプリ」から「OS標準より楽（貼って1タップ・長文一括・誤読矯正）」に再設定する。「無料・登録不要・無制限・非送信」は前提であって差別化にはならない。

誤読矯正辞書を差別化の主軸に格上げ
- SSML はブラウザで実質使えないため、テキスト置換辞書（speak() 前に JS で置換）が唯一の現実解。送信ゼロ・依存ゼロを崩さず実装できる。先行事例に TTSReader の Pronunciation Dictionary（日本語対応は未確認）。
- 対象は「読みが一意に決まる」語（英略語・固有名詞・記号）に限定。文脈依存の同音異義語（行った・重複・方）は誤爆するので対象外。
- 段階：案A でも英略語中心の軽量プリセット辞書を入れられる（コストほぼゼロ）。kuromoji.js（漢字読み分け）は初期ロード約18MB で軽量コンセプトと緊張するため案B以降の選択。Piper TTS（約70〜85MB）は将来の大改造扱い。
- 実装前に「誤読の実機洗い出し → 辞書化 → ビフォーアフター比較」を一度行い、効果を実証してから訴求に使う（現状は技術的合理性ベースの仮説）。

広告方針の確定
- リワード型は AdSense Offerwall（2025-06 GA）として正規に導入可能だが、「貼って1タップ」の手軽さと広告ゲートが衝突するため見送る（できないからではなく、合わないから選ばない）。
- 広告はアンカー広告（操作時表示・閉じられる）を主軸にする。自動 pop-up・強制 pre-roll は配置ポリシー違反。
- 収益は過大視しない。ツールサイトは RPM が構造的に低く、広告は運営費の足し程度に位置づける。
