# リサーチ：競合・読み精度・広告（2026-06-16）

- 調査担当：白石（並列リサーチ＋誤読辞書トピックの再取得）
- 目的：(1)棒読みアプリ競合に対する優位性 (2)誤読辞書での精度改善の可否 (3)リワード型広告のWeb導入可否
- 凡例：`[事実]`＝出典あり／`[事実・要原本照合]`＝二次情報で再確認推奨／`[仮説（裏未取得）]`＝推論／`[未確認]`＝取れず

## 結論サマリー

- 競合の見立てを修正：本当の競合はスマホアプリではなく OS標準の読み上げ機能。汎用アプリ無料音源・Web Speech API・OS標準は中身が同じ端末内蔵 TTS で音質に差がつかない。勝負どころは「OS標準より楽か」（貼って1タップ・長文一括・誤読矯正）。
- 誤読辞書（テキスト置換）は実装する価値が明確。SSML はブラウザで実質使えないため、置換辞書が唯一の現実解。送信ゼロ・依存ゼロのまま実装でき、英略語・固有名詞・記号に効く。同音異義語は対象外と割り切る。
- リワード型広告は AdSense Offerwall（2025-06 GA）として正規に導入可能。ただし「貼って1タップ」の手軽さと広告ゲートが思想的に衝突するため、採るならアンカー広告が現実解。

## 1. 競合：本当の相手はアプリでなく OS標準

- `[事実]` 棒読みちゃんは Windows 専用（スマホ版なし）・無料・最終更新 2020-12-24 で停滞。音源は AquesTalk/SAPI5。出典：作者公式 https://chi.usamimi.info/Program/Application/BouyomiChan/
- `[事実]` SofTalk は 2022年7月に AquesTalk 対応をライセンス問題で中止。AquesTalk 系のスマホ無料提供には構造的継続リスク。出典：窓の杜 https://forest.watch.impress.co.jp/docs/news/1427701.html
- `[事実]` ゆっくり棒読みトーク（iOS/Android）は完全無料・評価4.7（3.1万件）。AquesTalk 使用。ただしレビューに「文字数制限あり」「ケ→ガ」「ヴ→ア」の誤読報告。出典：App Store id542600107
- `[事実]` @Voice Aloud Reader（Android定番）は基本無料＋Premium。無料時の音源は端末内蔵 TTS＝Web Speech API と同じ。
- `[事実]` 汎用アプリ無料音源・OS標準（iOS「選択項目の読み上げ」/ Android「テキスト読み上げの出力」）・Web Speech API は同じ端末内蔵 TTS。Web版の実質的な最大競合は OS標準機能。出典：MDN Web Speech API ほか
- `[事実]` 「アプリを入れたくない層」の裏付け：スマホユーザー n=442 のうち約62%が「DL後7日以内にアプリを削除した経験あり」。若年層は登録/ログイン要求を嫌厭。出典：iRidge 2022 https://iridge.jp/news/202212/33343/
- `[仮説（裏未取得）]` 長文を単発でコピペして聞く用途はカジュアル利用＝ブラウザが選ばれやすい構造に近い（諸説あり・過大評価注意）。

導かれる方針：狙う層は妥当。ただし優位性を「対アプリ」でなく「OS標準より楽」に置く。OS標準は選択→メニュー→読み上げの数タップで、長文一括や誤読の手当てができない。「URL開く→貼る→1タップ→長文まるごと→誤読も矯正済み」の体験差が分水嶺。「無料・登録不要・無制限・非送信」の4点は前提（無料Web TTS群が一様に先行）であって、それ自体は差別化にならない。

## 2. 読み精度：誤読辞書（テキスト置換）が唯一の現実解

- `[事実]` SSML（phoneme / sub / say-as 等）は Web Speech API 仕様書に記載があるが、主要ブラウザ全てで実質未実装。Chrome（crbug #795371・macOS でタグ文字列が読み上げられる不具合）、Firefox（bugzilla #1425523・8年以上未解決）、Safari/WebKit（#122208）、Edge。出典：mdn/browser-compat-data Issue #15663
- `[事実]` 結論として SSML は「使えない」前提で設計する。代替はテキスト置換辞書のみ。
- `[事実]` テキスト置換辞書は先行事例あり：TTSReader が2026年初頭に「Pronunciation Dictionary」機能をリリース（TTS エンジンに渡す直前に「AI→エーアイ」のような置換を適用）。出典：https://ttsreader.com/tags/pronunciation/
  - `[未確認]` ただし TTSReader の辞書が日本語テキストに対応しているかは未確認（英語UI中心の可能性）。
- `[事実]` 実装は単純：SpeechSynthesisUtterance.text に渡す文字列を speak() 前に正規表現/replace で置換するだけ。ブラウザ依存も権限問題も発生しない。送信ゼロ・依存ゼロを完全に維持できる。
- `[事実]` 得意な範囲：英略語（AI・UI・API・URL）、固有名詞、専門用語、記号など「読みが一意に決まる」固定置換。競合の誤読報告（ケ→ガ、漢字を中国語に誤検出 等）の多くはこの類型で潰せる。
- `[事実＋仮説]` 苦手な範囲：文脈依存の同形異音語（行った＝いった/おこなった、重複、方）。単純置換は誤爆するので対象外にする。過剰置換（「AI」と「愛」等）は単語境界条件でリスクを下げるが完全には防げない。
- `[事実]` 精度をさらに上げる選択肢と限界：

  | 手段 | 精度向上範囲 | コスト | 依存 |
  |---|---|---|---|
  | テキスト置換辞書（手動） | 英略語・固有名詞・記号の固定誤読 | 低（実装数時間） | ゼロ |
  | kuromoji.js 補完 | 漢字読み分け・送り仮名 | 中（辞書 gzip 約18MB＋初期化0.8秒〜） | ゼロ |
  | Piper TTS（WASM） | 音質・読み精度を大幅向上 | 高（合計約70〜85MBの初期DL） | ゼロ（完全オフライン） |
  | VOICEVOX | 最高水準 | 高（ローカルサーバー起動が必要） | サーバー依存 |

- `[事実]` Chrome は単一 Utterance が約200〜250文字でカットされる（既存実装の分割キューイングで対応済み）。

導かれる方針：手動の置換辞書（英略語・固有名詞中心のプリセット）は、コストほぼゼロ・依存ゼロで最も費用対効果が高い差別化の芽。kuromoji.js は精度が上がるが初期ロード約18MB が「軽量ブラウザ完結」コンセプトと緊張関係。Piper は別次元の品質だが重く、将来の大改造扱い。

## 3. 広告：リワードは Offerwall で可能だが思想と衝突

- `[事実]` AdSense Offerwall が2025年6月に一般提供（GA）。来訪者が任意で短い広告（6〜30秒）を視聴するとコンテンツアクセスが一定時間（典型12時間）解放される。最低トラフィック/規模要件は公式ヘルプに明記なし＝個人/小規模でも標準ポリシーを満たせば導入できる建て付け。AMP 非対応。出典：https://support.google.com/adsense/answer/13865318
- `[事実]` Google Ad Manager も Web 向け rewarded 広告を提供するが中〜大規模向け。個人の現実解は AdSense Offerwall。出典：https://support.google.com/admanager/answer/9116812
- `[事実]` 報酬提供広告のポリシー：現金・仮想通貨など直接的金銭の報酬は禁止／明示的オプトイン必須／特定選択へ誘導する文言は禁止。出典：https://support.google.com/admanager/answer/7496282
- `[事実]` アンカー広告（画面端固定・ユーザーがページを操作した時に表示・閉じられる）が AdSense 公式機能。UX 侵襲が最小で単機能ツールの現実解として最有力。
- `[事実]` 避けるべき：意図しない自動 pop-up・強制 pre-roll は配置ポリシー違反。雑な常時インタースティシャルはモバイル SEO の負要素（ただし現行では CWV 以外の page experience は直接ランキング上昇に寄与せず、ペナルティの強弱は誇張しない）。
- `[事実・要原本照合]` 単機能・単一ページ・滞在短のツールサイトは RPM が構造的に低い。広告は「あれば足し」程度に見る。

導かれる方針：リワード（Offerwall）は「できないから諦める」ではなく「手軽さという売りに合わないから選ばない」。広告はアンカー広告を主軸にする。

## 検証で残った留保（実装前に必須）

白石の検証エージェントとリサーチの未確認項目から、実装前に確かめるべきこと。

1. 誤読の実機洗い出し → 辞書化 → ビフォーアフター比較。「誤読が起きる」は事実だが「辞書で直る」は未実証。主要音声（Google 日本語 / Microsoft Haruka / Apple）で実機確認。
2. 競合 Web TTS の辞書機能の日本語対応有無（TTSReader 等）を実調査。現状は「沈黙からの推論」。
3. Offerwall の挙動・適格性を最新ヘルプ/change log で再確認（ポリシーは年度改定される）。
4. スマホ実機での挙動（長文15秒停止の回避・画面ロック/バックグラウンド再生・Android WebView 非対応）。
5. 音声品質は端末内蔵 TTS 止まりで AquesTalk のキャラ性やクラウド AI 音声には構造的に及ばない。辞書や手軽さでは埋められない弱みとして認識する。
